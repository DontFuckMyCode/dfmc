# sc-nosqli — NoSQL Injection

**Date:** 2026-04-29
**Scope:** D:\Codebox\PROJECTS\DFMC
**Status:** NOT APPLICABLE — no NoSQL document/cache client; bbolt is KV with byte keys

## Verdict

No findings. DFMC has no MongoDB, Redis, Cassandra, DynamoDB, CouchDB, or Elasticsearch client. The only persistence layer is bbolt — a local key-value file with no query language. User-supplied input does not appear as a raw bbolt key in any path checked.

### 2. bbolt key construction review

bbolt is a pure KV store: keys are `[]byte`, values are `[]byte`, no operators, no expression language, no `$where`, no `eval`. The classic NoSQL-injection vectors (`$ne`, `$gt`, `$where`, JS evaluation in queries) **do not exist** in bbolt by construction. Risk is reduced to "can attacker make us write/read the wrong key?".

Bucket-key writes audited in [internal/storage/store.go](../internal/storage/store.go), [internal/memory/store.go](../internal/memory/store.go), [internal/taskstore/store.go](../internal/taskstore/store.go), [internal/drive/persistence.go](../internal/drive/persistence.go), and [internal/conversation/manager.go](../internal/conversation/manager.go):

| Bucket | Key source | Notes |
|---|---|---|
| `codemap_cache`, `meta` (storage) | Fixed string keys (`"schema_version"`, etc.) | No user input |
| `working_memory` | Fixed key `bucketWorkingKey` | No user input |
| `episodic_memory`, `semantic_memory` | `entry.ID` — generated by `memory.NewEntry`, not caller-controlled | [internal/memory/store.go:165](../internal/memory/store.go) |
| `tasks` | `t.ID` from `supervisor.Task` (UUID-shaped) | [internal/taskstore/store.go:41](../internal/taskstore/store.go) |
| `drive-runs` | `run.ID` (planner-generated) | [internal/drive/persistence.go:68](../internal/drive/persistence.go) |
| Conversations (JSONL files, not bbolt) | Conversation ID + timestamp | Filename sanitised to `[a-zA-Z0-9_-]` |

User-supplied strings never become a raw bbolt key in any path checked. The closest thing is `taskstore.Get(id)` and `drivestore.Get(id)` taking an HTTP path parameter — but the worst case is "look up a different existing task," not arbitrary key tampering, and the lookup is bounded by the bucket scope.

### 3. No `$`-prefix operators / no JS execution

NoSQL injection sinks (`bson.M{"$where": …}`, `redis.Eval`, Cassandra CQL parameter splicing, ES `query_string` injection) are absent because none of those clients are imported.

## Bottom line

sc-nosqli is not applicable to DFMC. bbolt is a flat KV store with byte keys, no query operators, and no expression evaluation. Key construction is reviewed and never feeds raw user input into a bucket key. If a future change adds Mongo/Redis/Cassandra/ES, re-run this scan.
