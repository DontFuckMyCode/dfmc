# sc-sqli + sc-nosqli Results

## Findings

### [Informational] SQL Injection — Not Applicable

- **Scope**: DFMC codebase
- **Reason**: DFMC has no SQL surface. The codebase uses only `go.etcd.io/bbolt` (embedded key-value store) for persistence. Zero SQL drivers, ORMs, or raw query strings exist anywhere in the production path.

**Verification:**

| Check | Result |
|---|---|
| `go.mod` SQL driver imports | None — only `go.etcd.io/bbolt v1.4.3` |
| `database/sql` usage | 0 matches |
| SQL query strings (`SELECT`, `INSERT`, etc.) | Only in security scanner test data and knowledge-base prose, not in runtime query construction |
| Third-party SQL clients (pgx, lib/pq, mysql, sqlite3, etc.) | 0 matches |

All bbolt access uses fixed bucket names and `[]byte` keys with no query language:

- `storage/store.go:98` — `tx.CreateBucketIfNotExists([]byte(bucket))` with fixed bucket names
- `memory/store.go:172` — `b.Put([]byte(entry.ID), data)` where ID is system-generated
- `taskstore/store.go:49` — `b.Put([]byte(t.ID), data)` where ID is a supervisor.Task field
- `drive/persistence.go:68` — `b.Put([]byte(run.ID), data)` where ID is planner-generated

### [Informational] NoSQL Injection — Not Applicable

- **Scope**: DFMC codebase
- **Reason**: No MongoDB, Redis, Cassandra, DynamoDB, Elasticsearch, or any other document/cache client. bbolt is a flat KV store with byte keys, no operators, no expression language, and no `$where` / JS evaluation. Key construction reviewed and never flows raw user input into bucket keys.

**Verification:**

| Check | Result |
|---|---|
| NoSQL client imports (mongodb, redis, gocql, elastic, etc.) | 0 matches |
| bbolt key injection vectors | None — bucket keys use fixed names or system-generated IDs |
| Query operator injection (`$ne`, `$gt`, `$where`, etc.) | Not applicable — bbolt has no operators |
| JS evaluation in queries | Not applicable — bbolt has no query language |

**Key source audit:**

| Bucket | Key source | User-controlled? |
|---|---|---|
| `meta`, `config`, `plugins`, `codemap_cache`, `ast_cache` | Fixed string constants | No |
| `memory_working` | Fixed key `"working"` | No |
| `memory_episodic`, `memory_semantic` | `entry.ID` — generated by `Store.Add` with microsecond timestamp + random suffix | No |
| `tasks` | `supervisor.Task.ID` (UUID-shaped, not caller-supplied) | No |
| `drive-runs` | `run.ID` — planner-generated | No |
| Conversations (file-backed JSONL) | System-generated IDs, validated with `validateConvID` (path traversal protection at `storage/store.go:468`) | No |

## No Issues Found

DFMC uses bbolt exclusively as a local embedded KV store. There is no SQL query surface, no NoSQL query surface, no query operator injection possible, and no raw user input used in key construction. The `validateConvID` function at `storage/store.go:468` additionally protects against path traversal in conversation file artifact paths.
